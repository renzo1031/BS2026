# 留守儿童帮扶平台项目规格

## 1. 项目定位

本项目是一套面向儿童关爱机构的业务协作平台，核心结果是让每一项帮扶需求从建档、审核、匹配、执行、验收到评价归档均可追踪。系统采用一套 Vue 应用、三种布局：

- 公开官网：平台介绍、服务类型、工作流程、公开统计、公益动态、合规说明和登录/注册入口。
- 志愿者门户：认证资料、脱敏需求大厅、我的申请、我的任务、回访和完成申请。
- 管理工作台：儿童档案、需求审核、志愿者匹配、服务验收、组织权限和审计。

公开层不展示任何可识别儿童的姓名、照片、学校、家庭地址、监护人联系方式或个案叙事。

## 2. 调研与选型结论

| 主题 | 结论 | 处理 |
| --- | --- | --- |
| 公益官网信息架构 | 中国儿童少年基金会、壹基金公开层均突出机构信息、公益项目、动态、参与入口和信息公开 | 采纳官网首页与信息公开区，但不引入在线捐赠 |
| 公开个案展示 | 与未成年人隐私和最小必要原则冲突 | 不采纳，需求大厅仅显示脱敏业务摘要 |
| Element Plus | 后台组件完整，但容易形成惯性模板风格 | 不采纳 |
| Naive UI | Vue 3 与 TypeScript 支持良好，视觉轻量 | 暂缓 |
| Ant Design Vue | 表格、表单、步骤流和企业工作台组件成熟，用户已明确指定 | 采纳，并通过 ConfigProvider Token 自定义主题 |
| TypeScript 7 | 当前 vue-tsc 3.3.7 仍依赖已被 TypeScript 7 移除的编译器子路径 | 暂缓，锁定 TypeScript 5.9.3 |
| 微服务、Redis、消息队列 | 1 人开发、低并发和单体事务场景没有收益 | 不采纳，使用模块化单体 |

调研来源：

- 中国儿童少年基金会：https://www.cctf.org.cn/
- 壹基金：https://www.onefoundation.cn/
- Ant Design Vue：https://github.com/vueComponent/ant-design-vue

## 3. 架构假设与质量目标

- 当前及 12 个月团队规模：1 名开发者。
- 交付频率：按项目里程碑每周交付。
- 产品性质：用户产品，含公开信息页与登录后业务系统，不是单独营销站。
- 一年规模假设：100 日活用户，p99 约 10 RPS。
- 部署预算：本地部署或单台低成本服务器，每月不超过 20 美元等值成本。
- 数据等级：含未成年人敏感个人信息，按高敏感数据处理。
- API 目标：p50 < 150ms、p95 < 500ms、p99 < 1000ms（20 并发、10000 条业务数据的本机基准）。
- 前端目标：移动 4G 下 LCP <= 2.5s、INP <= 200ms、CLS <= 0.1。
- 可用性目标：部署环境月度可用性 99.5%，项目验收以可复现启动和主流程通过为准。

## 4. 角色与数据范围

| 角色 | 数据范围 | 允许操作 | 禁止操作 |
| --- | --- | --- | --- |
| 系统管理员 | 全局 | 用户、部门、角色权限、志愿者认证、审计、全局统计 | 日常儿童档案和需求业务审批 |
| 部门主管 | 本部门 | 档案与需求审核、查看申请、匹配志愿者、服务验收 | 审核本人提交或其他部门数据 |
| 个案人员 | 本部门且本人负责 | 儿童建档、需求创建、跟踪任务、评价、归档 | 审核本人数据、跨部门访问 |
| 志愿者 | 公开脱敏需求及本人数据 | 认证、申请、执行任务、回访、提交完成 | 查看未分配儿童敏感信息、操作他人任务 |

权限由 RBAC、部门数据范围、本人数据范围和业务状态四层共同判断。前端菜单与按钮仅作体验控制，后端 Service 必须再次校验。

## 5. 状态机

### 儿童档案

`DRAFT -> PENDING_REVIEW -> ACTIVE | REJECTED -> ARCHIVED`

- 个案人员提交草稿；部门主管审核。
- 审核人不得等于提交人。
- 仅 ACTIVE 档案可创建帮扶需求。

### 志愿者认证

`UNVERIFIED -> PENDING_REVIEW -> APPROVED | REJECTED -> SUSPENDED`

- 志愿者提交认证；系统管理员审核。
- 仅 APPROVED 志愿者可申请需求。

### 帮扶需求

`DRAFT -> PENDING_REVIEW -> APPROVED | REJECTED -> MATCHED -> IN_PROGRESS -> PENDING_ACCEPTANCE -> COMPLETED -> CLOSED`

- 个案人员创建并提交，部门主管审核。
- APPROVED 后进入脱敏需求大厅。
- 接受志愿申请时，在同一事务内接受一条申请、拒绝其余申请、创建唯一有效任务并进入 MATCHED。
- 志愿者开始服务后进入 IN_PROGRESS；至少有一条回访才能提交完成。
- 主管验收后进入 COMPLETED；个案人员评价后进入 CLOSED，评价只能提交一次。
- DRAFT、PENDING_REVIEW、APPROVED、MATCHED 状态可按权限取消；取消匹配需求时同步终止任务。

## 6. 功能与页面

| 布局 | 页面 |
| --- | --- |
| 官网 | 首页、服务介绍、信息公开（首页区块）、登录、志愿者注册 |
| 志愿者门户 | 工作台、需求大厅、需求详情、我的申请、我的任务、任务详情、个人认证 |
| 管理工作台 | 工作台、儿童档案列表/详情、需求列表/详情、志愿者审核、用户与部门、角色权限、审计日志 |

首页首屏直接展示平台名称、服务定位和“了解服务/进入平台”入口；首屏下方露出服务类型。视觉使用真实业务语义的生成图片，不使用真实儿童素材。

## 7. 数据模型

核心表共 15 张：

`sys_department`、`sys_role`、`sys_permission`、`sys_role_permission`、`sys_user`、`auth_session`、`child_profile`、`volunteer_profile`、`aid_request`、`aid_request_review`、`aid_application`、`service_assignment`、`visit_record`、`service_feedback`、`audit_log`。

关键约束：用户名、部门编码、档案编号和需求编号唯一；同一志愿者不能重复申请同一需求；一项需求同时只能有一个有效任务；一项任务只能评价一次；关键业务表使用 version 乐观锁；状态操作使用事务；审计日志不可业务删除。

## 8. API 概要

- `/api/public/**`：官网公开统计与脱敏需求。
- `/api/auth/**`：注册、登录、退出、当前用户。
- `/api/users/me`、`/api/volunteers/me/**`：个人资料和志愿认证。
- `/api/children/**`：档案增改查、提交、审核、归档。
- `/api/aid-requests/**`：需求增改查、提交、审核、取消、申请列表。
- `/api/applications/**`：申请、撤回、接受。
- `/api/assignments/**`：任务查询、开始、回访、提交完成、验收、评价。
- `/api/admin/**`：用户、部门、角色权限、志愿审核、审计和统计。

响应统一为 `code`、`message`、`data`、`traceId`；列表使用 `items`、`total`、`page`、`size`。写接口使用 Bean Validation，业务错误使用稳定错误码。

## 9. 安全与隐私

- Spring Security + JWT + 服务端会话记录，退出和账号禁用立即使令牌失效。
- BCrypt 存储密码；连续 5 次失败锁定 15 分钟；错误信息不区分账号不存在或密码错误。
- 儿童姓名、监护人姓名/电话/地址和志愿者联系方式使用 AES-256-GCM 加密，密钥仅来自环境变量。
- SQL 全部参数化；生产 CORS 仅允许配置的可信来源。
- 志愿者认领前只能看到档案编号片段、年龄段、区域、类别、标题和需求摘要。
- 日志与审计不记录密码、令牌、密钥或明文个人信息。
- 预置数据均为虚构数据，不提供照片或证明材料上传。

## 10. 验收条件

- 四类角色的路由、按钮、接口权限和数据范围一致生效。
- 儿童建档、需求审核、志愿申请匹配、服务执行评价四条流程可完整闭环。
- 跨部门访问、越权操作、非法状态流转、重复申请、重复匹配和重复评价均被后端拒绝。
- 后端测试与编译、前端生产构建、MySQL 空库初始化和真实 HTTP 主流程通过。
- Playwright 验证官网、登录、核心操作及桌面/移动布局，且无未解释的 console error。
- README、SQL、默认账号和验证记录与实际项目一致，无未解决 P0/P1 问题。
