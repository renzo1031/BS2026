# 校园搭子平台项目规格

状态：已确认，开发事实源
确认日期：2026-07-11
题目：基于微信小程序的校园搭子平台设计与实现

## 1. 大白话定义

学生通过微信登录并绑定真实校园身份，自行发布任意合规的搭子场景。活动经本校审核员或平台管理员审核后公开，其他已认证学生可以申请加入；发起人审批申请，成员在活动工作区沟通并完成活动，之后进行结构化互评。平台提供举报、下架、封禁、申诉和审计闭环。

系统不是恋爱交友、陌生人私聊、交易或紧急救援平台。

## 2. 产品目标与约束

- 核心结果：缩短合规活动从发布到收到首个有效申请的时间，并提高活动完成率。
- 首期单校园运行，所有业务表保留 `campus_id` 数据边界。
- 默认服务 18 周岁及以上在校学生；真实上线需由校园身份源提供资格结论。
- 学生可以自定义场景，不使用固定业务分类白名单。
- 推荐排序必须可解释，只依据同校、标签、时间、地点、余量和发布时间，不使用 AI 或虚构匹配率。
- 不做支付、悬赏、跨校、连续定位、公开联系方式、关注/好友/朋友圈、视频、普通文件和商业 IM。
- 消息队列暂不采用；达到多实例、多消费者、积压、死信或回放需求时必须重新评审并征得用户同意。

## 3. 角色与权限

### 学生 `STUDENT`

- 数据范围：本校已发布活动、本人业务、已接受活动的成员与会话。
- 可以：维护资料、申请认证、发布活动、处理本人活动申请、参与、聊天、完成确认、评价、收藏、拉黑、举报和申诉。
- 不可以：读取其他申请人的记录、进入未参加会话、操作他人活动、跨校访问或审核本人内容。

### 校园审核员 `CAMPUS_REVIEWER`

- 数据范围：仅所属校园的认证、活动、举报和案件必要证据。
- 可以：认领并审核校园身份和活动，处理举报，下架内容，提出或执行账号限制，形成处置记录。
- 不可以：跨校审核、任意浏览聊天、修改学生原文、分配角色、删除审计记录。

### 平台管理员 `PLATFORM_ADMIN`

- 数据范围：全局校园配置、用户、审核、申诉和审计。
- 可以：正常参与审核，接管超时或升级案件，管理审核员与校园配置，处理申诉，查看审计日志。
- 不可以：静默覆盖历史结论、直接改写学生业务或物理删除审计记录。

鉴权必须在后端同时校验角色、`campus_id`、资源归属、账号状态和业务状态。前端菜单和按钮隐藏不能替代后端校验。

## 4. 校园身份绑定

流程：

`wx.login -> code2Session -> 受限账号 -> 选择校园和标识类型 -> 提交学号/校园唯一标识 -> 完成所有权证明 -> 自动或人工审核 -> 原子绑定`

- 微信 `openid` 仅证明微信侧身份，不证明在校资格。
- 仅本地微信开发者工具 `develop` 版使用固定测试 code `student-a` 复用已认证测试账号；`trial/release` 必须使用真实 `wx.login -> code2Session`，生产环境关闭 `WECHAT_DEV_LOGIN_ENABLED`。
- 仅输入正确学号不得通过，必须完成学校 SSO/CAS/API、校园邮箱验证码或人工证明中的一种。
- 本项目提供可替换的校园验证适配器；无真实校方接口时使用测试名册和人工审核，不宣称已接入真实校方数据。
- 标识规范化后使用 HMAC-SHA-256 指纹去重；确需保留的原值使用 AES-GCM 加密，界面只显示脱敏值。
- 唯一约束：`(campus_id, identifier_type, identifier_fingerprint)`。
- 换绑需原账号确认或进入申诉流程，所有行为写入审计日志。
- 认证证明存 MinIO 私有桶，仅有身份审核权限的审核者可通过短期签名 URL 查看，并按保留策略删除。

## 5. 活动字段和规则

- `scene_name`：学生自定义场景，2-20 字。
- `title`：5-50 字。
- `description`：20-1000 字。
- `tags`：最多 5 个自定义标签，每个 2-12 字，随活动审核。
- `meeting_mode`：`ONLINE/OFFLINE/HYBRID`，不是固定场景分类。
- `start_at/end_at/apply_deadline`：必须满足申请截止不晚于开始、结束晚于开始。
- `public_location`：广场可见的概略地点。
- `member_location_detail`：仅审核者和已接受成员可见。
- `capacity`：2-50，发起人计入或不计入必须统一按实现定义；首期按参与者名额，不含发起人。
- `join_requirement`：最多 500 字。
- `join_questions`：最多 3 个问题，不实现动态表单引擎。
- `media_ids`：最多 6 张审核通过的活动图片。
- `campus_id/creator_id`：仅从认证主体写入，拒绝客户端提供。

已发布但尚无接受成员的活动修改核心字段后重新审核并暂时隐藏；已有接受成员时核心字段锁定，只允许取消并克隆为新草稿。

## 6. 状态机

### 认证

`PENDING -> APPROVED | REJECTED`，`APPROVED -> EXPIRED | REVOKED`。

### 账号

`ACTIVE -> LIMITED | SUSPENDED -> ACTIVE | CLOSED`。被限制或封禁后旧令牌立即失效，但保留申诉能力。

### 活动审核

`NOT_SUBMITTED -> PENDING -> APPROVED | REJECTED | WITHDRAWN`。

审核认领不是业务状态。认领保存 `reviewer_id/claim_expires_at/version`；条件更新保证并发只有一个认领者，冲突返回 `409 REVIEW_ALREADY_CLAIMED`。

### 活动生命周期

`DRAFT -> RECRUITING -> IN_PROGRESS -> COMPLETION_PENDING -> COMPLETED`，分支为 `CANCELLED/EXPIRED`。

### 活动治理

`NORMAL -> UNDER_REVIEW -> REMOVED`，与审核及生命周期分开。

### 申请

`PENDING -> ACCEPTED | REJECTED | WITHDRAWN`，`ACCEPTED -> CANCELLED`。

- 创建者不能申请自己的活动。
- 同一用户同一活动只能申请一次。
- 接受申请必须在事务内原子检查状态与余量，禁止超员。
- 重复接受、取消和退出必须幂等，人数只能增减一次。

### 活动完成

成员完成状态为 `PENDING -> CONFIRMED | DISPUTED | AUTO_CONFIRMED`。全员确认或超时且无人异议后活动完成；异议自动生成治理案件。
活动进入完成确认时写入 `completion_deadline_at`，维护任务超时后自动确认仍为 `PENDING` 的成员，并在无异议时完成活动。

### 会话

`OPEN -> READ_ONLY | LOCKED -> CLOSED`。首个申请被接受后开放，只有有效成员可以访问；活动完成或取消后只读，治理案件可锁定。

### 举报与申诉

`SUBMITTED -> REVIEWING -> ACTIONED | DISMISSED -> APPEALED -> UPHELD | REVOKED`。
举报认领不是业务状态。认领保存 `assignee_id/claim_expires_at/version`；维护任务会释放过期认领，允许其他审核员继续处理。

## 7. 图片与对象存储

使用 S3 兼容对象存储抽象，开发环境为 MinIO，不使用 Base64 或 BLOB 保存图片。

支持：头像、活动图片、聊天图片、认证证明、举报证据。首期不支持视频、压缩包和普通文件。

上传流程：

`wx.uploadFile -> 后端流式接收 -> 大小/MIME/魔数/像素校验 -> 安全重编码并清除 EXIF -> MinIO 隔离区 -> 微信 mediaCheckAsync -> APPROVED/REJECTED -> 授权访问`

- 头像最大 2 MB；活动和聊天图片最大 5 MB；认证和举报证据最大 8 MB。
- 活动最多 6 张图片；聊天每条最多 1 张。
- 仅允许 JPEG、PNG、WebP；拒绝 SVG、动图和伪造 MIME。
- WebSocket 不传图片二进制，只发送审核通过的 `fileId`。
- 桶保持私有。普通内容使用短期签名 URL，证据仅案件授权审核者可访问。
- 对象键使用随机 UUID；数据库保存对象键、所有者、业务归属、SHA-256、MIME、大小、宽高、状态和生命周期。
- 定时清理未绑定、审核失败和超过保留期的对象。
- MinIO Community Server 固定 `RELEASE.2025-10-15T17-29-55Z` 从源码构建；Java SDK 固定 `8.6.0`。该版本与 Spring Boot 3.5.9 的 Jackson 2.19 依赖线兼容；暂不采用会引入 Jackson 2.21 和 Okio 3.16 的 SDK 9.0.3。

## 8. WebSocket 与 Redis

- 小程序使用单个 `SocketTask` 连接 `WSS /ws`。
- 客户端先通过 REST 获取 60 秒、单次使用的握手票据，避免把 JWT 放入 URL。
- 心跳、指数退避重连、`clientMessageId` 幂等和消息游标补拉必须实现。
- 消息先校验成员资格、会话状态和内容安全，再写 MySQL；事务提交后才 ACK 和推送。
- MySQL 是消息历史事实来源，断线后通过 REST 补拉。
- Redis 只保存握手票据、在线状态、限频、失效令牌、微信 access token 和短期幂等键，不保存业务真相。
- 首期单实例不使用 Redis Pub/Sub、Streams 或其他消息队列。

## 9. 个人中心

- 总览：头像、昵称、认证徽标、待审核/待参与/待评价、未读消息和近期活动。
- 资料：头像、简介、兴趣标签、年级/专业及可见范围。
- 我的活动：发布、申请、参与记录，按业务状态筛选和分页。
- 收藏：收藏活动及失效提示。
- 信誉评价：完成、取消和结构化评价分布，不提供不透明信用分。
- 身份与安全：脱敏身份、续期、换绑、退出、注销。
- 安全中心：举报、申诉和黑名单。
- 设置：站内及微信订阅消息、隐私可见范围和授权管理。

公开资料不得包含完整学号、真实姓名、证明文件、联系方式、精确轨迹或最后在线时间。

## 12. API 约定

- 基础路径 `/api/v1`，约 45 个 REST API 和一个 `/ws` WebSocket 入口。
- 成功响应：`{ code, message, data, requestId, timestamp }`；没有业务数据时也返回 `data: null`，不省略该字段。
- 分页参数：`page/size`，`size` 最大 50。
- 使用明确 HTTP 状态：400、401、403、404、409、429、500。
- 时间存储为 UTC，响应使用 ISO 8601 带时区格式。
- 状态动作使用受限命令枚举，客户端不得直接赋值状态字段。
- 所有查询参数化；所有写入做 Bean Validation 和业务校验。
- 关键写操作支持幂等键或状态条件更新。

核心资源组：认证与资料、校园绑定、活动、申请和成员、消息与通知、文件、评价与收藏、举报申诉、审核治理、校园配置与审计。

活动管理接口补充：

- `GET /api/v1/operations/activities`：管理员分页查看活动，支持 `keyword`、`reviewStatus`、`lifecycleStatus`、`moderationStatus`、`meetingMode` 筛选；平台管理员全局可见，校园审核员按校园范围查询，学生禁止访问。
- `GET /api/v1/operations/activities/{id}`：管理员只读活动完整链详情，包含活动状态、发起人、素材、报名申请、参与成员、活动状态记录及关联举报/处置/审计时间线。

## 13. 技术栈

- 小程序：原生微信小程序、TypeScript，开发基础库 `3.16.2`，最低 `2.32.3`。
- 管理端：Vue `3.5.39`、Vite `8.1.4`、TypeScript `5.9.3`、Pinia、Vue Router、Axios、Element Plus `2.14.3`。TypeScript 7.0.2 暂不采用，原因是当前 `vue-tsc 3.3.7` 尚未适配其内部模块导出变化。
- 后端：Java 17、Spring Boot `3.5.9`、Spring Security、MyBatis-Plus `3.5.16`、Spring WebSocket、MinIO Java `8.6.0`。
- 数据：MySQL `8.0.45`、Redis `8.6.2`、MinIO Community Server `RELEASE.2025-10-15T17-29-55Z`。
- 架构：模块化单体，按业务模块组织，Controller 只处理协议，规则与权限在 Service 层。
- 本地基础设施：使用本机 MySQL、Redis 和 MinIO；仓库不提供容器化部署方式。

## 14. 非功能目标

假设：当前和一年内团队规模均为 1 人；按日开发、阶段发布；用户端面向学生，后台面向内部运营；低成本单机部署，每月基础设施预算不超过 300 元。

- API：`p50 < 150ms`、`p95 < 500ms`、`p99 < 1s`，不含微信外部接口耗时。
- 管理端移动 4G：`LCP < 2.5s`、`INP < 200ms`、`CLS < 0.1`。
- 单实例可用性目标：`99.5%`。
- 所有秘密从环境变量注入；仓库只提交 `004-backend/env.example`。
- 管理员密码使用 BCrypt；JWT 设置过期，退出、封禁和角色变化立即失效。
- 生产 CORS、HTTPS/WSS 和微信合法域名使用显式白名单。
- 日志不记录令牌、密钥、完整校园标识、证明 URL 或敏感个人信息。
- 管理后台和小程序覆盖加载、空数据、失败、无权限和不存在状态。

## 15. 验收标准

1. 空库可完整执行 SQL，后端测试与编译通过，管理端生产构建通过。
2. 微信登录、管理员正确/错误密码、未登录、过期令牌和封禁令牌经过验证。
3. 仅知道他人学号不能绑定；两个微信账号并发绑定同一身份只有一个成功。
4. 未配置固定场景时，学生仍能发布自定义活动；场景、标签和图片审核前不可见。
5. 本校审核员不能审核其他校园；管理员可以按规则接管；并发认领和决策只有一个成功。
6. 学生不能读取或修改他人活动、申请、会话、认证证明和举报证据。
7. 容量为 2 时并发接受 20 个申请，最终严格只有 2 个成功。
8. 非法状态流转、重复审批、重复退出、重复完成和重复评价均被拒绝或幂等处理。
9. WebSocket 验证成员权限、心跳、重连、重复消息、断线补拉和会话锁定。
10. 文件上传拒绝超限、伪造 MIME、SVG 和越权删除；聊天图片审核前不推送。
11. 举报、下架、封禁、申诉和审计形成完整链路，被举报人看不到举报人身份。
12. 个人中心统计与活动、申请、评价真实数据一致。
13. MySQL、Redis、MinIO 使用本机服务完成连接验证，配置项与 `004-backend/env.example` 一致。
14. 小程序在微信开发者工具和真实测试账号条件允许范围内验证；没有真实 AppID 的能力必须标记未验证。
15. 无未解决 P0/P1 权限、安全、数据损坏或主流程问题。

## 16. 明确未验证依赖

- 真实学校 SSO/CAS/学籍接口和成年资格字段尚未提供。
- 正式认证小程序 AppID、可选服务类目、订阅消息模板、内容安全权限和发布审核资质尚未提供。
- 因此开发阶段提供可替换网关和测试适配器，最终交付不得把模拟能力宣称为真实校方或微信线上接入。

## 17. 前端实施边界与组件图

管理后台采用鉴权后的 Vite SPA，主要设备为校园审核人员桌面浏览器，兼顾 720px 以上平板；不依赖 SEO。性能门为移动 4G p75 `LCP < 2500ms`、`INP < 200ms`、`CLS < 0.1`，初始 JS 预算 200KB gzip、单个懒加载路由增量 80KB gzip；Lighthouse 性能不低于 85、可访问性不低于 90。可访问性目标为 WCAG 2.2 AA，当前一人项目由项目开发者负责。

管理端组件边界：

- `App`：只承载跳转链接与路由出口。
- `AdminLayout`：侧栏、顶栏、当前用户与退出；不持有业务列表状态。
- `PageHeader`、`AsyncState`、`StatusTag`：分别负责页头、加载/空/失败状态和状态语义显示。
- `usePagedRequest`：统一分页请求、加载、错误与刷新副作用。
- 活动、身份、举报、文件审核各自使用“路由容器 + 数据表 + 详情/决策抽屉”，表格通过 props 接收数据并通过 emits 上报选择，抽屉通过 props 展示对象并上报认领/决策。
- 用户与审核员、校园、推荐标签配置使用平台管理员路由；页面直接调用后台管理接口，保存后依赖后端审计和令牌版本失效保证配置可追踪。
- `auth` Pinia store：唯一登录态来源；路由守卫负责页面角色，按钮指令仅改善体验。

小程序按主包登录/认证/广场/详情/消息/个人中心和业务、账户两个分包组织；每页只组合业务组件，网络、会话、WebSocket 与分页状态分别放入独立工具或状态模块。小程序以中低端安卓、校园 4G 为主要设备，不渲染不可信 HTML，不在本地保存校园标识原文或证明 URL。
