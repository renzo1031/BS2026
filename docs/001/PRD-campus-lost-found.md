# 校园失物招领中心 PRD

## 1. 文档定位

本 PRD 面向开发实现，不是论文、PPT 或客户展示稿。项目目标是开发一个真实可运行的校园失物招领中心，默认技术栈为 Spring Boot + MyBatis 或 MyBatis-Plus + MySQL + Vue 3。

本 PRD 采用“先判断必要性，再确定范围”的方式：每个模块都必须说明为什么需要、完整做到什么程度、不做什么、缺少什么会导致业务断链。

## 2. 调研依据与取舍结论

### 2.1 调研依据

- 公开校园失物招领毕设项目普遍包含用户发布、分类浏览、失物/招领管理、管理员审核、公告或留言等能力，说明“前台门户 + 用户中心 + 后台管理”是合理系统形态。
- MyBatis-Plus 是 MyBatis 的增强工具，适合快速实现 CRUD、分页和条件查询；本项目可在 MyBatis 与 MyBatis-Plus 中二选一，但同一项目内保持统一。
- OWASP Logging Cheat Sheet 强调应用日志应服务安全和业务追溯，日志事件需要能回答 when、where、who、what；因此后台审计不能只是写入日志表，还要能在物品详情和后台日志页查询。
- OWASP Password Storage 与 Spring Security 文档均强调密码应使用专门的密码哈希算法或 PasswordEncoder，不允许明文或简单 MD5。

参考链接：

- https://github.com/bsxj02/lost-and-found-system

### 2.2 总体取舍结论

本项目不做成单纯信息发布网站，也不做成只有管理员 CRUD 的后台。失物招领的真实闭环至少包含：

1. 拾到物品后发布招领，后台审核，上架展示，用户认领，核验通过，线下交接，归档。
2. 丢失物品后发布寻物，后台审核，上架展示，其他用户提交线索，发布人确认有效，归档。
3. 管理员或保管员对物品进行审核、保管、交接、下架、归档，并保留完整操作日志。

## 3. Problem Statement

校园内失物招领通常分散在班级群、公众号、宿管处、保卫处或线下登记本中，存在信息不集中、重复发布、认领过程不可追踪、管理员无法核验责任链的问题。用户不知道物品当前状态，管理员也难以追溯谁发布、谁认领、谁审核、谁修改、最后一次更新时间和每次状态变化原因。

## 4. Solution

建设一个校园失物招领中心，包含前台门户、用户个人中心、保管员处理端和管理后台。

系统支持用户发布失物/招领信息、浏览搜索、提交认领申请、提交线索反馈、查看本人记录；保管员负责实物保管、认领核验、交接登记；管理员负责审核上架、用户与分类管理、公告管理、全量物品查询、审计日志追踪和数据统计。

## 5. 用户角色

### 5.1 普通用户

面向学生、老师或校内工作人员。可以注册登录、发布失物/招领、提交认领申请、提交线索、查看本人发布和本人申请。

必要性判断：必须有。没有普通用户端，系统会退化成后台录入工具，不符合校园真实使用场景。

### 5.2 物品保管员

面向失物招领处、宿管处、保卫处或服务中心工作人员。负责接收实物、登记保管位置、核验认领材料、记录交接结果。

必要性判断：推荐有。校园失物招领存在“信息发布”和“实物保管”两个环节，保管员能让认领闭环更真实。若项目规模需要压缩，可由管理员兼任，但 PRD 默认保留该角色。

### 5.3 管理员

负责系统配置、内容审核、用户管理、分类地点管理、公告管理、全量数据查看、日志审计和异常处理。

必要性判断：必须有。没有管理员，发布审核、违规内容处理、数据追踪和权限管理无法闭环。

## 6. 权限边界

| 角色 | 可查看数据 | 可新增 | 可修改 | 可审核/处理 | 可删除/归档 | 审计可见性 | 明确禁止 |
|---|---|---|---|---|---|---|---|
| 普通用户 | 公开上架物品、本人发布、本人认领、本人线索、本人通知 | 失物、招领、认领申请、线索反馈 | 本人草稿、被驳回内容、未进入交接的本人申请 | 本人线索确认、本人申请取消 | 本人草稿删除、本人记录申请关闭 | 仅本人业务时间线 | 查看他人手机号、查看后台日志、强制修改状态 |
| 物品保管员 | 已审核通过且需保管/交接的物品、认领申请、交接记录 | 保管记录、交接记录、核验备注 | 保管位置、交接备注、认领核验结果 | 认领核验、交接确认、物品归还 | 不可物理删除，可提交异常关闭 | 可看自己处理的业务日志 | 管理用户角色、删除用户、修改系统配置 |
| 管理员 | 全部用户、全部物品、全部申请、全部线索、全部日志 | 分类、地点、公告、管理员备注 | 全部业务数据的管理字段、审核备注、状态纠偏 | 发布审核、认领复核、异常下架、归档 | 可逻辑删除/下架/归档，必须填写原因 | 全量审计日志和物品时间线 | 物理删除关键业务数据、篡改历史日志 |

后端服务层是权限最终来源，前端菜单和按钮只做体验层控制。

## 7. 状态模型

### 7.1 物品状态

```text
草稿 -> 待审核 -> 已上架 -> 待认领核验 -> 待交接 -> 已完成
              -> 已驳回
              -> 已下架
              -> 已归档
```

状态规则：

- 草稿：仅发布人可编辑和删除。
- 待审核：发布人可撤回，管理员可通过或驳回。
- 已上架：前台可见，用户可认领或提交线索。
- 待认领核验：存在认领申请，保管员或管理员核验。
- 待交接：核验通过，等待线下交接。
- 已完成：完成交接或寻回确认，普通用户不可再编辑。
- 已驳回：发布人可修改后重新提交。
- 已下架：管理员因违规、重复、过期等原因下架。
- 已归档：长期完成或关闭记录，保留审计，不再参与前台搜索。

### 7.2 认领申请状态

```text
待审核 -> 已通过 -> 待交接 -> 已领取
       -> 已驳回
       -> 已取消
```

### 7.3 线索反馈状态

```text
待确认 -> 有效 -> 无效 -> 已关闭
```

## 8. 核心业务流程

### 流程一：拾物招领认领闭环

1. 普通用户发布拾到物品信息，填写分类、地点、时间、图片、特征描述和联系方式。
2. 系统生成物品编号，状态为待审核。
3. 管理员审核内容，通过后上架，驳回时必须填写原因。
4. 其他用户浏览或搜索物品，提交认领申请，填写证明材料。
5. 保管员或管理员核验认领申请。
6. 核验通过后进入待交接，线下交接时记录领取人、证件后四位或学工号、交接地点、交接时间、经办人。
7. 交接完成后物品状态为已完成，认领申请状态为已领取。
8. 全流程写入时间线和操作日志。

### 流程二：失物登记线索反馈闭环

1. 普通用户发布寻物信息，描述遗失时间、地点、物品特征、悬赏说明可选。
2. 管理员审核后上架。
3. 其他用户提交线索，说明发现地点、时间、联系方式和图片。
4. 发布人查看线索并标记有效或无效。
5. 发布人确认找回后关闭寻物记录。
6. 管理员可处理恶意线索、重复线索和长期未关闭记录。

### 流程三：后台审核与审计追踪闭环

1. 管理员进入后台物品管理，查看所有物品，不受本人数据限制。
2. 管理员可按状态、分类、地点、发布人、认领人、审核人、更新时间筛选。
3. 管理员进入物品详情，查看发布信息、认领申请、线索反馈、图片、交接记录和完整时间线。
4. 管理员对待审核内容执行通过、驳回、下架、归档、状态纠偏等操作，必须填写原因。
5. 每次操作写入日志，记录操作者、角色、目标对象、前后状态、请求来源、操作结果和时间。
6. 管理员可在操作日志页按用户、对象、动作、时间、结果查询。

## 9. 功能模块决策表

| 模块 | 是否必要 | 为什么需要 | 完整性要求 | 不该做什么 | 如果缺少会怎样 |
|---|---|---|---|---|---|
| 登录注册与账号安全 | 必须 | 用户发布、认领和审计都需要身份 | 注册、登录、退出、当前用户、密码哈希、角色识别 | 不做明文密码，不做无登录认领 | 无法追责，后台看不到谁发布谁认领 |
| 前台门户与搜索 | 必须 | 用户需要发现物品和寻物信息 | 分类、地点、关键词、状态筛选，详情页 | 不做纯静态首页，不堆营销内容 | 用户无法使用，系统像后台工具 |
| 物品发布与审核 | 必须 | 防止虚假、违规、重复信息直接公开 | 草稿、提交、审核、驳回、上架、下架 | 不允许发布即公开且无审核 | 信息质量不可控 |
| 认领申请与核验 | 必须 | 招领业务必须确认物品归属 | 证明材料、申请状态、核验备注、交接 | 不做“一键认领成功” | 业务没有真实闭环 |
| 线索反馈 | 必须 | 寻物业务不能只发布信息，还要有人反馈 | 线索提交、有效/无效确认、通知 | 不做无状态留言板 | 寻物流程无法完成 |
| 保管交接 | 必须 | 实物交付是失物招领关键环节 | 保管位置、经办人、交接时间、领取确认 | 不要求线上支付、物流配送 | 认领后无法证明是否交付 |
| 通知公告 | 推荐 | 用户需要知道审核、认领、线索结果 | 站内通知、公告发布、已读状态 | 不做复杂消息推送平台 | 用户需要主动反复刷新 |
| 后台管理 | 必须 | 管理员要管全量业务数据 | 物品、申请、线索、用户、分类、地点、公告 | 不做只有统计卡片的假后台 | 数据失控，无法答辩说明管理能力 |
| 审计日志 | 必须 | 需要追踪谁做了什么 | 操作日志、对象时间线、前后状态、原因 | 不允许日志只写不查 | 出问题无法追责 |
| 基础统计 | 推荐 | 毕设后台需要体现管理价值 | 总量、待审核、待交接、完成率、分类分布 | 不做大屏炫技 | 后台缺少管理视角 |

## 10. 后台管理详细要求

### 10.1 管理员物品列表必须展示

- 物品编号。
- 类型：失物、招领。
- 标题。
- 分类。
- 发现/遗失地点。
- 发现/遗失时间。
- 当前状态。
- 发布人姓名、账号、手机号。
- 当前认领人姓名、账号、手机号，若无则显示无。
- 审核人姓名、审核时间、审核结果。
- 当前保管员姓名，若无则显示无。
- 创建时间。
- 最后更新时间。
- 最后操作人。
- 最后操作摘要。

必要性判断：必须。后台如果只能看到标题和状态，管理员无法判断责任链，也无法处理争议。

### 10.2 管理员物品详情必须包含

- 基础信息：标题、描述、分类、地点、时间、图片、联系方式。
- 发布信息：发布人、提交时间、发布来源、是否本人撤回过。
- 审核信息：审核人、审核时间、审核结论、驳回原因或通过备注。
- 认领信息：申请人、申请时间、证明材料、核验结果、驳回原因。
- 线索信息：提交人、线索内容、图片、有效性、确认人。
- 保管交接：保管位置、保管员、交接地点、交接时间、领取确认、经办人。
- 状态时间线：从创建到归档的所有状态变化。
- 操作日志：所有新增、修改、审核、下架、交接、归档、异常处理记录。

### 10.3 审计日志字段

操作日志至少包含：

- 日志 ID。
- 操作人 ID、姓名、角色。
- 操作对象类型：物品、认领申请、线索、用户、分类、公告。
- 操作对象 ID。
- 操作动作：创建、提交、审核通过、审核驳回、修改、下架、认领申请、核验通过、核验驳回、交接完成、归档、登录失败等。
- 操作前状态。
- 操作后状态。
- 操作结果：成功或失败。
- 失败原因或业务原因。
- 请求 IP。
- User-Agent。
- 请求路径。
- 创建时间。

不做内容：不做日志篡改、物理删除日志、普通用户查看全量日志。

## 12. 数据库表设计概要

默认 13 张核心表：

1. `sys_user`：用户账号、姓名、手机号、学号/工号、密码哈希、状态。
2. `sys_role`：普通用户、保管员、管理员。
3. `sys_user_role`：用户角色关联。
4. `sys_menu`：后台菜单和权限标识。
5. `lf_item`：失物/招领主表，包含状态、发布人、审核人、认领人、保管员、最后操作人、最后更新时间。
6. `lf_item_image`：物品图片。
7. `lf_category`：物品分类。
8. `lf_location`：校园地点。
9. `lf_claim_application`：认领申请。
10. `lf_clue_feedback`：线索反馈。
11. `lf_custody_handover`：保管与交接记录。
12. `sys_notice`：公告和站内通知。
13. `sys_operation_log`：操作日志和审计记录。

必要字段规则：

- 所有业务表必须包含 `created_by`、`created_at`、`updated_by`、`updated_at`、`deleted`。
- 关键业务表必须包含 `status`。
- 审核类字段必须包含 `reviewer_id`、`review_time`、`review_result`、`review_reason`。
- 交接类字段必须包含 `handler_id`、`handover_time`、`handover_location`、`receiver_id`、`remark`。

## 13. 接口设计概要

### 13.1 认证与用户

1. `POST /api/auth/register` 注册。
2. `POST /api/auth/login` 登录。
3. `POST /api/auth/logout` 退出。
4. `GET /api/auth/me` 当前用户。
5. `PUT /api/users/me` 修改个人资料。
6. `PUT /api/users/me/password` 修改密码。

### 13.2 前台物品

7. `GET /api/items` 前台物品列表。
8. `GET /api/items/{id}` 物品详情。
9. `POST /api/items` 发布失物/招领。
10. `PUT /api/items/{id}` 编辑本人草稿或驳回内容。
11. `POST /api/items/{id}/submit` 提交审核。
12. `POST /api/items/{id}/cancel` 撤回或关闭本人记录。
13. `GET /api/users/me/items` 我的发布。

### 13.3 认领与线索

14. `POST /api/items/{id}/claims` 提交认领申请。
15. `GET /api/users/me/claims` 我的认领。
16. `POST /api/items/{id}/clues` 提交线索。
17. `GET /api/users/me/clues` 我的线索。
18. `POST /api/clues/{id}/confirm` 发布人确认线索有效或无效。

### 13.4 保管员处理

19. `GET /api/staff/claims` 待核验认领列表。
20. `POST /api/staff/claims/{id}/approve` 核验通过。
21. `POST /api/staff/claims/{id}/reject` 核验驳回。
22. `POST /api/staff/items/{id}/handover` 交接完成。
23. `PUT /api/staff/items/{id}/custody` 更新保管信息。

### 13.5 管理后台

24. `GET /api/admin/items` 全量物品列表。
25. `GET /api/admin/items/{id}` 后台物品详情。
26. `POST /api/admin/items/{id}/review` 审核通过或驳回。
27. `POST /api/admin/items/{id}/offline` 下架。
28. `POST /api/admin/items/{id}/archive` 归档。
29. `GET /api/admin/claims` 全量认领申请。
30. `GET /api/admin/clues` 全量线索。
31. `GET /api/admin/users` 用户管理列表。
32. `PUT /api/admin/users/{id}/status` 启用或禁用用户。
33. `GET /api/admin/categories` 分类列表。
34. `POST /api/admin/categories` 新增分类。
35. `PUT /api/admin/categories/{id}` 修改分类。
36. `GET /api/admin/locations` 地点列表。
37. `POST /api/admin/locations` 新增地点。
38. `PUT /api/admin/locations/{id}` 修改地点。
39. `GET /api/admin/notices` 公告列表。
40. `POST /api/admin/notices` 发布公告。
41. `GET /api/admin/logs` 操作日志。
42. `GET /api/admin/statistics` 基础统计。

业务量说明：接口数量超过 25 个参考线，但每个接口都服务具体页面或状态流转。实现时可以按模块拆 Controller，避免单个 Controller 过大。

## 14. User Stories

1. 作为普通用户，我想注册并登录系统，以便发布和认领物品。
2. 作为普通用户，我想修改个人资料，以便保持联系方式准确。
3. 作为普通用户，我想按关键词、分类、地点搜索物品，以便快速找到相关信息。
4. 作为普通用户，我想查看物品详情和图片，以便判断是否是我的物品。
5. 作为普通用户，我想发布拾到物品信息，以便帮助失主找回。
6. 作为普通用户，我想发布遗失物品信息，以便让其他人提供线索。
7. 作为普通用户，我想保存草稿，以便信息没填完时稍后继续。
8. 作为普通用户，我想查看审核驳回原因，以便修改后重新提交。
9. 作为普通用户，我想提交认领申请，以便证明物品归属。
10. 作为普通用户，我想查看认领申请进度，以便知道是否通过核验。
11. 作为普通用户，我想取消错误的认领申请，以便避免影响他人处理。
12. 作为普通用户，我想提交线索反馈，以便帮助失主寻找物品。
13. 作为普通用户，我想确认线索是否有效，以便关闭无效线索。
14. 作为普通用户，我想在找回物品后关闭记录，以便前台不再展示。
15. 作为普通用户，我想查看我的发布、我的认领、我的线索，以便管理自己的记录。
16. 作为普通用户，我想收到审核、认领、线索、交接通知，以便及时处理。
17. 作为保管员，我想查看待核验认领申请，以便处理线下领取。
18. 作为保管员，我想查看物品保管位置，以便找到实物。
19. 作为保管员，我想记录认领核验结果，以便留下处理依据。
20. 作为保管员，我想登记交接记录，以便证明物品已领取。
21. 作为管理员，我想审核发布内容，以便防止虚假、违规或重复信息上架。
22. 作为管理员，我想查看所有物品，以便全局管理失物招领数据。
23. 作为管理员，我想查看每个物品是谁发布、谁认领、谁审核、谁处理，以便追责。
24. 作为管理员，我想按状态、分类、地点、发布人、认领人、审核人、更新时间筛选，以便定位问题记录。
25. 作为管理员，我想查看完整操作日志，以便追踪每次关键操作。
26. 作为管理员，我想下架违规或过期记录，以便保持前台信息质量。
27. 作为管理员，我想归档已完成记录，以便保留历史但不影响前台搜索。
28. 作为管理员，我想管理分类和地点，以便规范用户发布内容。
29. 作为管理员，我想管理公告，以便通知校内用户流程或制度变化。
30. 作为管理员，我想禁用异常账号，以便处理恶意发布或刷线索行为。
31. 作为管理员，我想查看基础统计，以便了解待审核、待交接、已完成数量。

## 15. Implementation Decisions

- 系统采用前台门户、个人中心、保管处理端、管理后台四个功能区域，但保管处理端可以复用后台布局并通过角色权限控制菜单。
- 后端采用 Spring Boot + MyBatis 或 MyBatis-Plus + MySQL，项目内统一选择一种数据访问方案。
- 密码使用 PasswordEncoder 或 BCrypt 进行安全哈希存储。
- 权限采用角色权限 + 数据归属 + 状态校验组合，不依赖前端隐藏按钮完成权限控制。
- 物品、认领申请、线索反馈、保管交接都必须有状态字段和状态流转校验。
- 管理后台物品列表必须是全量视角，个人中心列表必须是本人视角。
- 审计日志独立建表，同时在物品详情中展示对象时间线。
- 删除采用逻辑删除、下架或归档；关键业务数据和日志不做物理删除。
- 图片上传先按本地文件或静态资源路径实现，不引入对象存储。

## 16. Testing Decisions

测试应覆盖外部行为，不测试内部实现细节。

必须验证：

- 注册后密码不是明文存储。
- 普通用户只能修改本人草稿或驳回记录。
- 普通用户不能访问后台全量物品接口。
- 管理员可以查看所有物品并筛选发布人、认领人、审核人。
- 待审核物品只有管理员能审核。
- 已完成物品不能被普通用户继续编辑。
- 认领申请状态不能越级跳转。
- 交接完成后物品和认领申请状态同步变化。
- 每次审核、驳回、下架、交接、归档都生成操作日志。
- 管理员能在物品详情看到完整时间线。

## 17. Out of Scope

本期不做：

- 微信小程序、App、公众号推送。
- 在线支付、悬赏支付。
- 物流配送。
- OCR 自动识别物品。
- 人脸识别、身份证识别。
- 多校区多租户复杂组织架构。
- 大屏可视化。
- 复杂工作流引擎。
- 物理删除关键业务数据和日志。

## 18. 缺口自检

### 已覆盖

- 登录注册和密码安全。
- 前台门户。
- 个人中心。
- 管理后台。
- 物品发布审核。
- 认领申请核验。
- 线索反馈。
- 保管交接。
- 分类地点管理。
- 通知公告。
- 操作日志和时间线。
- 3 类角色权限边界。
- 3 条主业务闭环。

### 仍需在开发前细化

- 物品图片上传大小、格式和数量限制。
- 手机号、学号/工号是否唯一。
- 学生和教师是否共用普通用户角色。
- 线下交接是否需要上传签收照片。
- 默认测试账号和初始化数据。
- 前端 UI 主题和组件库细节。

### 明确不该补

- 不为了“显得高级”加入 AI 识别、地图定位、实时聊天或大屏。
- 不把保管员扩展成多个复杂部门角色，除非用户后续明确要多组织。
- 不做只有管理员能录入数据的后台系统。

## 19. 验收标准

- 用户可以注册、登录、退出，密码安全哈希存储。
- 普通用户可以发布失物/招领、提交认领、提交线索、查看本人记录。
- 管理员可以审核发布内容，能通过、驳回、下架、归档。
- 保管员可以核验认领申请并登记交接。
- 后台物品列表能看到发布人、认领人、审核人、保管员、最后更新时间和最后操作人。
- 物品详情能看到发布、审核、认领、线索、交接、归档的完整时间线。
- 操作日志页能按用户、对象、动作、时间、结果查询。
- 前台只展示已上架且未归档的记录。
- 个人中心只展示本人相关记录。
- 关键接口有后端权限校验，不能只靠前端隐藏按钮。
