# AI 项目开发流程规范

本文档给 AI 使用，用于本仓库内 Spring Boot + Vue/React 毕设项目的完整开发流程。核心原则：**先设计，再计划，再开发，再验证**。

`xxx` 表示项目代号，例如 `001`、`002`、`mall`。

## 与其他规则的关系

- 本文档是**完整的端到端操作手册**，详细说明每一步做什么、怎么做、如何验证
- `rules/project-workflow.md` 是**精简的流程规则框架**，定义核心原则和关键决策点
- 本文档在关键步骤引用专项 rules 和 skills，避免重复维护实现细节

## AI 使用规则

- 用户只要求某一步时，只做该步骤
- 用户说”开始””继续””按计划做”时，按本文档顺序进入下一步
- 未完成 PRD、`backend-design-spec`、`frontend-design-spec` 和 `development-plan` 前，不进入正式开发
- 不把项目做成 Demo，不把业务系统默认写成纯后台 CRUD 模板
- 初始化业务数据必须落库，不写死在前端、后端、配置常量或 Mock 接口
- 措辞遵循 `rules/wording-standards.md`，代码、文档、数据库脚本禁止出现"演示、示例、测试数据、Demo、Mock"等词汇
- 默认不提交 Git，除非用户明确要求
- 文档、规则、计划类改动不需要编译验证
- 前端执行 `npm install`、`npm run build` 后，必须按规则清理对应 `node_modules`

## 总流程

```text
第 0 步：读取规则与上下文
  -> 第 1 步：确定项目代号和目录
  -> 第 2 步：确认产品边界和角色
  -> 第 3 步：编写 PRD
  -> 第 4 步：编写 backend-design-spec
  -> 第 5 步：编写 frontend-design-spec
  -> 第 6 步：制定 development-plan
  -> 第 7 步：后端开发
  -> 第 8 步：后端接口测试
  -> 第 9 步：前端开发
  -> 第 10 步：前后端联调
  -> 第 11 步：文档同步
  -> 第 12 步：交付检查
```

## 标准目录

标准目录结构详见 `rules/common-conventions.md`。

核心约定：

```text
workspace/
├── xxx-backend/          # 后端源码
├── xxx-frontend/         # 前端源码
└── docs/
    └── xxx/              # 项目文档
        ├── PRD.md
        ├── backend-design-spec.md
        ├── frontend-design-spec.md
        ├── development-plan.md
        ├── api-test.md
        └── database.md
```

## 第 0 步：读取规则与上下文

目标：明确当前规则、项目状态、用户最新要求。

必须读取：

- `AGENTS.md`
- `rules/skills.md`
- 当前任务相关规则：`rules/project-workflow.md`、`rules/backend.md`、`rules/backend-design-standards.md`、`rules/frontend.md`、`rules/frontend-design-standards.md`、`rules/common-conventions.md`、`rules/wording-standards.md`、`skills/windows-local-dev`
- 已存在的 `docs/xxx/` 文档
- 用户最新消息

产出：

- 明确当前要执行哪一步
- 明确是否已有项目代号
- 明确是否已有 PRD、设计规范、开发计划

禁止：

- 未读规则直接开发
- 忽略用户最新要求

## 第 1 步：确定项目代号和目录

目标：统一项目代号和目录边界。

参考：`rules/common-conventions.md`、`skills/product-architecture`

必须做：

- 确认项目代号 `xxx`
- 创建或确认 `xxx-backend/`、`xxx-frontend/`、`docs/xxx/`
- 多前端应用也使用同一项目代号扩展，例如 `xxx-admin/`、`xxx-staff/`（是否拆多前端参考 `skills/product-architecture`）

产出：

- 项目代号
- 标准目录

禁止：

- 文档散落到根目录
- 后端、前端文件混放
- 因为角色多就机械拆多个前端

## 第 2 步：确认产品边界和角色

目标：先确认产品形态，再写 PRD。

参考：`skills/product-architecture`

### 需求挖掘（AI 必须主动执行）

用户给出项目主题后，AI 不能直接开干，必须主动问 3-5 个澄清问题：

#### 需求挖掘问题模板

**1. 谁用这个系统？**
- 示例：二手交易平台
- AI 要问：是校园内的学生交易？还是面向所有人的？
- 区别决定：是否需要学号验证、实名认证、物流对接

**2. 核心场景是什么？**
- 示例：图书管理系统
- AI 要问：是公共图书馆？还是学校图书馆？
- 区别决定：办证流程、押金管理、学期催还

**3. 有什么特殊规则？**
- 示例：请假系统
- AI 要问：几级审批？病假事假流程一样吗？
- 区别决定：单级/多级审批、流程引擎

**4. 数据量大概多少？**
- 示例：商品管理
- AI 要问：预计多少商品？多少用户？
- 区别决定：是否需要分页、缓存、搜索优化

#### 执行流程

1. **AI 提出 3-5 个澄清问题**
2. **用户回答**（如果回答不清楚，AI 给出建议）
3. **AI 给出具体的业务场景建议**
   - 示例："我建议做单校+实名+线下交易+纯信息发布，这样业务完整（发布、浏览、下单、确认、评价），有多角色协作（买家、卖家、管理员），有状态流转，有权限设计，业务量达标。你看这样可以吗？"
4. **用户确认后开始写 PRD**

必须确认：

- 项目类型
- 主要用户和管理员
- 每类角色的目标、任务、数据范围、权限
- 是否存在公开门户、用户中心、工作人员工作台、管理员后台、移动端
- 是否需要单前端、多布局或多前端
- 是否真的需要工作台

产出：

- 用户角色清单
- 应用边界判断
- `Now / Later / Omit` 范围
- 前端应用形态结论

禁止：

- 把普通用户放进后台工作台
- 把管理页面当成整个产品
- 用假统计凑工作台
- 用户给个主题就直接写 PRD（必须先问澄清问题）

## 第 3 步：编写 PRD

目标：定义产品要做什么。

参考：
- `skills/enterprise-business-analysis` - 用于完整的业务分析（资源、生命周期、状态、操作、权限、审计、异常）
- `rules/common-conventions.md` - 通用约定

默认产出：

```text
docs/xxx/PRD.md
```

PRD 必须包含：

- 项目概述
- 用户角色
- 产品架构和应用边界
- 功能需求
- 状态流转
- 权限边界
- 数据库设计
- API 设计
- 项目结构
- 默认账号
- 当前范围、暂缓范围、不做范围

**关键：使用 `enterprise-business-analysis` skill 进行深度业务分析**

对于每个核心业务对象，必须分析：
- Business Object（业务对象）
- Lifecycle（生命周期）
- State（状态）
- Action（操作）
- Permission（权限）
- Data Scope（数据范围）
- Field Permission（字段权限）
- Audit（审计）
- Exception（异常处理）

### 业务闭环强制检查（AI 必须执行）

每个项目必须有至少 1 个核心业务闭环，包含：

#### 1. 多角色协作（至少 3 种角色）

**错误示例：**
- ❌ 只有管理员增删改查
- ❌ 只有普通用户和管理员，没有业务协作

**正确示例：**
- ✅ 二手交易：买家、卖家、管理员（买家发布 → 卖家浏览下单 → 买家确认发货 → 卖家确认收货 → 双方评价）
- ✅ 请假系统：员工、直属领导、HR、管理员（员工提交 → 领导审批 → HR备案 → 员工查看结果）
- ✅ 失物招领：普通用户、工作人员、管理员（用户发布 → 工作人员审核 → 用户认领 → 工作人员确认）

#### 2. 状态流转（至少 5 个状态）

**错误示例：**
- ❌ 只有"正常"和"删除"状态
- ❌ 只有"启用"和"禁用"状态

**正确示例：**
- ✅ 订单：待支付 → 已支付 → 待发货 → 已发货 → 已完成 → 已评价
- ✅ 文章：草稿 → 待审核 → 审核通过/驳回 → 已发布 → 已下架
- ✅ 失物：草稿 → 待审核 → 已发布 → 已认领 → 已归还 → 已关闭

#### 3. 数据关联（至少 3 层关联）

**错误示例：**
- ❌ 用户表、商品表各自独立，没有关联

**正确示例：**
- ✅ 用户 → 订单 → 商品 → 评价（用户有订单，订单有商品，商品有评价，评价属于用户）
- ✅ 用户 → 失物 → 认领记录 → 消息（用户发布失物，失物有认领记录，认领产生消息）

#### 4. 业务量要求

**必须达到：**
- ✅ 数据库：10-15 张表（不含 user、role 等系统表）
- ✅ 后端接口：20-30 个（不含 login、logout 等通用接口）
- ✅ 前端页面：15-20 个（不含 404、登录页）

**不能超过：**
- ❌ 数据库：>20 张表（太复杂，单人做不完）
- ❌ 后端接口：>50 个（太多，单人做不完）
- ❌ 前端页面：>30 个（太多，单人做不完）

**AI 必须主动检查：**
```
当前业务量统计：
- 数据库表：12 张 ✅
- 后端接口：25 个 ✅
- 前端页面：18 个 ✅
- 核心业务流程：2 个 ✅
```

如果不达标，AI 主动提出：
- "当前只有 X 张表、Y 个接口，建议增加 XX 模块"

如果超标，AI 主动提出：
- "当前有 X 张表，建议将 XX 模块标记为 Later"

### 权限矩阵强制检查（AI 必须执行）

PRD 必须包含【角色×功能】矩阵表：

**示例（二手交易平台）：**

| 功能 | 游客 | 买家 | 卖家 | 管理员 |
|------|------|------|------|--------|
| 浏览商品列表 | ✅ | ✅ | ✅ | ✅ |
| 查看商品详情 | ✅ | ✅ | ✅ | ✅ |
| 搜索商品 | ✅ | ✅ | ✅ | ✅ |
| 发布商品 | ❌ | ❌ | ✅ | ✅ |
| 编辑自己的商品 | ❌ | ❌ | ✅ | ✅ |
| 删除自己的商品 | ❌ | ❌ | ✅ | ✅ |
| 下架他人商品 | ❌ | ❌ | ❌ | ✅ |
| 购买商品 | ❌ | ✅ | ✅ | ✅ |
| 查看自己的订单 | ❌ | ✅ | ✅ | ✅ |
| 查看所有订单 | ❌ | ❌ | ❌ | ✅ |
| 评价商品 | ❌ | ✅ | ✅ | ❌ |
| 删除他人评价 | ❌ | ❌ | ❌ | ✅ |

**检查规则：**
1. ✅ 高权限角色必须包含低权限角色的所有能力
   - 管理员能审核，也要能浏览、查看、发布
2. ✅ 不能出现"管理员只能审核，不能查看"
3. ✅ 数据权限必须明确（自己的/全部的/公开的）

**AI 必须主动验证这个表的逻辑一致性。**

验收：

- PRD 能指导后端设计
- PRD 能指导前端设计
- PRD 能解释完整业务闭环
- 权限矩阵表清晰且逻辑一致
- 业务量达标（10-15 张表、20-30 个接口、15-20 个页面）

禁止：

- 只写 Dashboard、用户管理、角色管理、菜单管理
- 留下 `business-domain`、`module` 这类泛化占位
- 把初始化数据写死在代码里
- 只说"管理员可以XX"而不说明权限边界、数据范围、字段权限
- 业务量不足就进入开发阶段

## 第 4 步：编写 backend-design-spec

目标：先设计后端，再写开发计划。

参考：
- `rules/backend.md` - 后端核心规则
- `rules/backend-design-standards.md` - 后端设计强制标准（数据库命名、API 规范、安全、性能、日志）
- `skills/feature-completeness` - 业务完整性检查
- `rules/common-conventions.md` - 通用约定

默认产出：

```text
docs/xxx/backend-design-spec.md
```

必须确认：

- 后端技术栈：Spring Boot、Java 版本、MySQL、MyBatis 或 MyBatis-Plus、JWT/Spring Security
- 后端包结构和分层职责
- 数据库名、表设计原则、初始化脚本位置
- 默认账号和初始化业务数据来源
- 统一返回、异常处理、参数校验
- 认证、权限、账号状态、Token 过期
- 核心业务状态流转
- API 分组和核心接口清单
- 消息、日志、统计是否属于当前阶段
- 后端验证策略

建议结构：

```text
1. 后端总体结论
2. 技术栈
3. 包结构
4. 数据库设计原则
5. API 设计原则
6. 认证与权限
7. 业务状态流转
8. 模块接口清单
9. 异常处理与校验
10. 验证策略
```

验收：

- 后端开发前已经知道怎么建包、建表、写接口、做权限和状态流转
- 后续 `development-plan` 能直接引用本设计拆后端阶段

禁止：

- 一边写后端代码一边临时决定 ORM、安全方案和包结构
- 只有 CRUD，没有状态流转和权限边界

## 第 5 步：编写 frontend-design-spec

目标：先设计前端，再写开发计划。

参考：
- `rules/frontend.md` - 前端核心规则
- `rules/frontend-design-standards.md` - 前端设计强制标准（配色、首页设计、视觉细节、响应式）
- `skills/frontend-architecture-conformance` - 前端架构一致性
- `skills/vue-graduation-project-standard` - 毕设项目规模标准
- `rules/common-conventions.md` - 通用约定

默认产出：

```text
docs/xxx/frontend-design-spec.md
```

必须确认：

- 前端应用形态：单前端、单前端多布局、多前端
- 每类角色进入哪个布局
- UI 组件库
- 主题风格、主色、圆角、字体、页面密度（按 `frontend-design-standards.md` 选业务主色，禁止蓝白默认配色）
- 前台页面清单（首页按业务类型设计，禁止表格首页）
- 后台页面清单
- 路由默认跳转
- 页面状态标准（loading、empty、error、validation、submitting、success）
- 视觉细节（卡片阴影、圆角、悬停、骨架屏、空状态插图、按钮图标）
- 响应式要求（首页、列表页、详情页至少手机端可用）

建议结构：

```text
1. 前端总体结论
2. 产品气质
3. 技术与组件库
4. 主题 Token
5. 布局策略
6. 前台页面清单
7. 后台页面清单
8. 组件使用规则
9. 页面状态标准
10. 路由与默认跳转
11. 建议目录结构
12. 后续实现检查清单
```

验收：

- 前端开发前已经知道布局、页面、组件库、风格和路由
- 前台不能简略，必须覆盖真实用户业务
- 后台不能只是通用模板，必须覆盖真实审核和管理流程

禁止：

- 未确认 UI 和布局就创建页面
- 普通用户页面被后台布局吞掉
- 工作台使用假数据

## 第 6 步：制定 development-plan

目标：在 PRD、后端设计、前端设计都完成后，再制定开发计划。

默认产出：

```text
docs/xxx/development-plan.md
```

必须包含：

- 总体开发顺序
- 当前进度
- 计划审查结论
- 包结构/目录约定
- 已完成内容
- 后端阶段
- 后端接口测试阶段
- 前端阶段
- 前后端联调阶段
- 收尾检查项

计划要求：

- 必须引用或吸收 `PRD.md`、`backend-design-spec.md`、`frontend-design-spec.md` 的关键结论
- 阶段按业务闭环拆，不按文件数量拆
- 每阶段有目标、状态、计划能力、验收标准、实现说明
- 已完成内容必须有证据
- 暂缓能力必须明确记录

禁止：

- 设计文档未完成就写开发计划
- 计划写成泛泛的任务列表
- 用户只要求计划时擅自开发

## 第 7 步：后端开发

目标：按 `backend-design-spec` 和 `development-plan` 实现后端。

参考：
- `rules/backend.md` - 后端核心规则
- `rules/backend-design-standards.md` - 后端设计强制标准（安全、校验、日志、性能）
- `skills/feature-completeness` - 业务完整性检查
- `skills/windows-local-dev` - Windows 本地开发环境

执行顺序：

1. 基础架构：`pom.xml`、`application.yml`、启动类、统一返回、异常处理、安全配置
2. 数据库：`init.sql`、表结构、索引、默认账号、初始化数据
3. Entity / DTO / VO
4. Mapper
5. Service：业务逻辑、状态校验、权限边界、异常处理、消息和日志
6. Controller：REST API、参数校验、统一返回

验收：

- 后端可启动
- 数据库初始化脚本可执行
- 核心接口覆盖 PRD 业务闭环
- 状态非法和权限失败有明确错误

禁止：

- 只有登录接口就说认证完成
- 状态流转只靠前端控制
- 初始化数据写死在代码里

## 第 8 步：后端接口测试

目标：验证后端真实业务链路。

参考：`skills/windows-local-dev` - Windows 本地开发环境

默认记录：

```text
docs/xxx/api-test.md
```

优先验证：

- 数据库初始化
- 默认账号登录
- 当前用户
- 核心业务新增、列表、详情
- 状态流转
- 权限失败
- 异常输入
- 消息、日志、统计

可用命令：

```powershell
mysql -uroot -proot
mvn test
```

验收：

- 至少跑通一条完整核心业务链路
- 权限失败和非法状态已验证
- 测试结果已记录

## 第 9 步：前端开发

目标：按 `frontend-design-spec` 和 `development-plan` 实现前端。

参考：
- `rules/frontend.md` - 前端核心规则
- `rules/frontend-design-standards.md` - 前端设计强制标准（配色、首页、视觉细节、响应式）
- `skills/frontend-architecture-conformance` - 前端架构一致性
- `skills/vue-graduation-project-standard` - 毕设项目规模标准
- `skills/windows-local-dev` - Windows 本地开发环境

执行顺序：

1. 创建基础工程文件
2. 配置 Vite、入口、全局样式和 UI 库
3. 封装 Axios、Token 注入、错误处理
4. 创建用户状态 store
5. 创建路由和权限守卫
6. 创建布局
7. 创建菜单和状态配置
8. 实现登录、注册、当前用户、退出
9. 实现前台门户和用户中心
10. 实现后台工作台和管理页面
11. 补齐页面状态和响应式

验收：

- 前端可启动或构建
- 角色登录后跳转正确
- 页面接真实 API
- loading、empty、error、validation、submitting、success、权限、响应式状态完整

禁止：

- 使用 Mock 业务数据
- 前台页面做得过于简略
- 为简单表单、表格、弹窗强行拆组件
- 执行依赖安装或构建后遗留 `node_modules`

## 第 10 步：前后端联调

目标：按真实业务流程验证前后端。

优先验证：

- 登录成功和失败
- 刷新后保持登录态
- 退出登录
- 公开列表和详情
- 新增、编辑、删除或关闭
- 状态流转按钮
- 审核通过、驳回、确认等动作
- 表单校验
- 后端错误提示
- 权限不足
- 无数据状态
- 移动端基础布局

问题定位顺序：

1. 前端请求参数
2. 后端接口返回
3. 数据库初始化数据
4. 权限或 Token
5. 页面状态展示

验收：

- 至少跑通一条完整端到端核心业务链路
- 关键角色流程可用
- 错误提示清楚

## 第 11 步：文档同步

目标：让文档和实际代码一致。

必须同步：

- `docs/xxx/PRD.md`
- `docs/xxx/backend-design-spec.md`
- `docs/xxx/frontend-design-spec.md`
- `docs/xxx/development-plan.md`
- `docs/xxx/api-test.md`
- `docs/xxx/database.md`

同步内容：

- 阶段状态
- 已完成内容
- 验证依据
- API 测试结果
- 数据库默认账号和初始化数据
- 暂缓与不做范围

禁止：

- 用”完成了”掩盖未做能力
- 文档和实际接口、数据库、目录结构冲突

## 第 12 步：交付检查

目标：交付前确认结果、验证、未做范围和临时产物。

参考：`rules/project-workflow.md` 第 9 节 - 交付说明与验证、Git、文档、注释规则

必须检查：

- 后端核心接口是否验证
- 前端核心页面是否验证
- 前后端关键流程是否联调
- 文档状态是否同步
- 暂缓和不做内容是否说明
- 措辞是否符合 `rules/wording-standards.md`（全局搜索"演示、示例、测试数据、demo、mock"，例外见该文档）
- 是否误留 `node_modules`、`dist`、临时文件、本机路径

最终回复必须说明：

- 完成了什么
- 修改了哪些关键文件
- 运行了什么验证
- 哪些能力暂缓或未做
- 如果执行过前端依赖安装或构建，是否已清理 `node_modules`
- 如果未验证，说明原因

## 阻断条件

遇到以下情况必须暂停开发，先补文档或确认：

- 没有项目代号
- 没有 PRD
- 没有 `backend-design-spec`
- 没有 `frontend-design-spec`
- 没有 `development-plan`
- 产品边界不清楚
- 前端应用形态不清楚
- 用户角色和权限边界不清楚
- 数据库初始化数据来源不清楚
- 要执行破坏性操作但目标路径未确认
